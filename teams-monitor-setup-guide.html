🤖 Teams Monitor

AI agents that live in your Microsoft Teams channels, powered by GitHub Copilot CLI and Agency.

Built with Agency + Copilot CLI + Custom MCP

What Is This?

Teams Monitor gives you persistent AI agents that live in your Microsoft Teams channels. Each agent runs as a long-lived GitHub Copilot CLI session connected to a specific working directory, with access to your full MCP toolkit. Agents respond to messages in-thread with styled Adaptive Cards, track conversation threads for follow-ups, maintain persistent memory across restarts, and run scheduled background tasks.

💬

Chat in Teams

Post a message in your Teams channel. A persistent AI agent picks it up within seconds (3s when active, 15s idle) and responds in-thread with a styled Adaptive Card.

🧠

Persistent Context

Each agent runs as a long-lived Copilot session in a specific working directory. No cold starts. Full conversation context is maintained across messages.

🤖

Adaptive Cards

Responses render as styled cards with a "Teams Monitor" header and channel subtitle, visually distinct from your own messages.

📝

Agent Memory

Each agent maintains a persistent memory file of decisions, learnings, and context that survives session restarts and is committed to git.

🔄

Thread Tracking

Agents monitor reply threads for 24 hours. Follow-up in a thread and the agent continues the conversation. Threads auto-close after 24h of inactivity.

⏰

Background Tasks

Schedule automated tasks like declining meetings outside work hours or summarizing email. Configurable via JSON.

Why Not Squad?

We evaluated Squad seriously

This project began as a Squad implementation. We installed @bradygaster/squad-cli, created agent charters following Squad conventions, and explored Squad's watch/triage/loop modes. We wanted to use it. The question was whether Squad's multi-agent orchestration layer added value for our specific use case: monitoring Teams channels and responding to messages.

What Squad excels at

Squad is a multi-agent dev team orchestrator for GitHub Copilot. It shines when you need:

  • A coordinator routing GitHub issues to specialist agents (frontend, backend, tester)
  • Cross-agent decision governance and shared decision logs
  • Persistent agent memory and history across sessions
  • Sprint ceremonies, agent casting, and team identity management
  • Watch mode that triages GitHub issues and dispatches Copilot agents

What we actually needed

Our problem is fundamentally different: single-agent-per-channel, responding to Teams chat messages in real time. There's no cross-agent coordination needed (each channel is independent). There's no GitHub issue triage. The "routing" is trivially solved by channel identity. Adding Squad's coordinator on top would be overhead without benefit.

🧠 Validated by independent Opus 4.7 architecture review

To confirm we weren't making a mistake, we ran a rigorous architectural review using Claude Opus 4.7 (highest available intelligence tier) comparing our solution against Squad. Key findings:

  • "Squad doesn't do this." Squad is a GitHub-issue-shaped orchestrator. Forcing it to monitor Teams channels would mean writing all the same plumbing we already wrote, then bolting Squad's coordinator on top to route work that doesn't need routing.
  • "Comparing them is a category error." The relevant question isn't "is our solution better than Squad" but "did we reinvent things Squad already solved well?" Answer: a few patterns worth borrowing (persistent agent history, structured decision logs), but not the runtime.
  • "Stop calling it Squad-inspired if it isn't." The review recommended honest attribution: cite Squad as prior art we evaluated, not as a dependency or inspiration we didn't actually use.

The same review also identified critical bugs in our bridge MCP (race conditions on shared state, message deduplication gaps, destructive queue reads) which we fixed before shipping.

Bottom line

Squad is excellent for what it does. It just doesn't do what we need. Our solution is purpose-built for Teams channel monitoring with the minimum viable stack: Agency for Copilot session management, a custom Node.js MCP for Teams integration, and PowerShell for orchestration. No unnecessary abstractions.

    -Argument '-NoProfile -ExecutionPolicy Bypass -File ".\start-agents.ps1" -AutoUpdate' `

$settings = New-ScheduledTaskSettingsSet -AllowStartIfOnBatteries `
    -DontStopIfGoingOnBatteries -ExecutionTimeLimit ([TimeSpan]::Zero) `
    -RestartCount 3 -RestartInterval (New-TimeSpan -Minutes 1) -MultipleInstances IgnoreNew
$principal = New-ScheduledTaskPrincipal -UserId "$env:USERDOMAIN\$env:USERNAME" `
    -LogonType Interactive

Register-ScheduledTask -TaskName "TeamsMonitor" -TaskPath "\" `
    -Action $action -Trigger $trigger -Settings $settings `
    -Principal $principal `
    -Description "Teams Monitor - Persistent Copilot agents with auto-update" -Force

5 Auto-Update

Teams Monitor can keep itself up to date automatically.

How it works

Every ~30 minutes, the monitor runs git fetch and compares your local HEAD against the remote. If new commits are detected:

  • With -AutoUpdate: Pulls changes, reinstalls bridge dependencies, kills all sessions, and restarts with the new code. Zero downtime (restart takes ~15 seconds).
  • Without -AutoUpdate: Logs a message suggesting you pull manually. No automatic changes.
PowerShell 7
# Run with auto-update enabled
.\start-agents.ps1 -AutoUpdate

# Or register the startup task with auto-update
# (see Step 6 above — the -AutoUpdate flag is included)

7 Troubleshooting

Teams MCP times out

The Teams MCP can be slow. The monitor retries automatically. If persistent, re-run auth.cmd to refresh OAuth tokens.

Duplicate replies

The bridge uses per-channel last-seen files and message ID deduplication. If you see duplicates, check for multiple running instances (the lock file should prevent this). Kill stale copilot/node processes and restart.

"Permission denied" on Teams MCP calls

Ensure you're running with the --yolo flag (already configured in all scripts). This auto-approves MCP permission requests in non-interactive mode.

No pretty TUI output

Use PowerShell 7 (pwsh), not Windows PowerShell 5.1. The Copilot CLI TUI requires a modern terminal.

OAuth token expired

Tokens last ~90 days. When expired, run auth.cmd interactively to re-authenticate.

Agent sessions keep recycling

This is normal. The LLM occasionally decides it's "done" despite the daemon prompt. The monitor auto-restarts sessions within 5 seconds. Messages are not lost (the bridge queues them). You'll see "session ended. Recycling..." in the log.

Thread replies not detected

The bridge only tracks threads that the agent has previously replied to. If you reply in an old thread from before the monitor started, post a new top-level message instead. Threads auto-close after 24 hours of inactivity.

9 Agent Memory

Each agent maintains a persistent memory file at .agents/memory/{agent-id}.md. After completing significant work (not routine polling), the agent appends a timestamped entry recording what it did, decisions it made, and context it learned.

Why this matters

Copilot sessions are ephemeral. When a session recycles (which happens periodically), the agent loses all in-session context. The memory file bridges that gap: each new session reads the file on startup and picks up where the last one left off. Over time, agents accumulate repo-specific knowledge, user preferences, and operational patterns.

Memory files are committed to git, so they travel with the repo. If you clone the project on a new machine, your agents already know what they've done.

10 Design Journey

This architecture evolved through several iterations, each solving a specific limitation.

1

Started with Squad + polling loop

Initial approach: use Squad's watch mode with a PowerShell loop that spawned a new agency copilot per message. Worked, but 90+ second latency (30s poll + 60s copilot cold start) and no context between messages.

2

Direct MCP HTTP proxy for fast polling

Discovered the Agency Teams MCP can run as a local HTTP server. Direct JSON-RPC calls poll channels in ~2.5 seconds instead of spawning a full copilot session. Reduced detection latency from 3 minutes to 30 seconds.

3

Custom Node.js MCP bridge

Built a per-channel MCP server that handles polling, message queuing, Adaptive Card formatting, self-DM notifications, and thread tracking. The bridge runs as a stdio MCP for each Copilot session, keeping the LLM focused on processing rather than plumbing.

4

Persistent sessions instead of per-message spawning

Each channel gets a long-lived Copilot session that calls check_messages() in a loop. No cold start, full context continuity, agent memory across interactions.

5

Browser watcher for near-instant detection

Graph API has no push notifications for local clients. Pure polling meant 15s latency. Solution: a persistent Playwright browser reads the Teams sidebar unread badges every 2 seconds (without clicking channels, which would mark them as read). When a badge appears, the bridge switches to 3s fast polling. Detection dropped from 15s to ~2-3s with no external infrastructure.

First attempt used channel-clicking, but Opus 4.7 review caught that this marks messages as read (UX regression) and DOM virtualization causes false positives. Rewritten to use sidebar badge reads only.

6

Opus 4.7 architecture reviews

At each stage, independent reviews by Claude Opus 4.7 validated the design, caught critical bugs (race conditions, duplicate replies, auth failures, DOM issues), and confirmed that Squad was the wrong tool for this specific problem. The reviews shaped the final architecture significantly.

Key insight: Graph API limitations drove every design decision

Microsoft Graph API cannot push-notify local clients (requires public HTTPS endpoint) and cannot mark messages as unread (no API exists). These constraints forced the three-tier architecture: browser watcher for fast detection, bridge MCP for message handling, persistent Copilot sessions for processing. Each tier works around a specific platform limitation without requiring external infrastructure.
